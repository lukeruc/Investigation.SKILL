# 讨论记录

## 原始文件存储

**问题**：调查中获取的原始文件（如审计报告 PDF、合同扫描件、裁判文书下载件）如何存储和关联？

当前模型只处理"从网页提取信息 → 实体/边入库"的路径。但某些信息（审计报告、PDF、付费数据库导出件）的价值在原文——任何二次转述都会损失信息（表格、附注、审计意见措辞）。

**讨论方向**：
- 案件目录下加 `evidence/` 存放原始文件
- 图中建 `document` 节点指向文件，source_chain 记录来源
- 子 Agent 需要文件下载和存储的指令

## 无头浏览器接入

**问题**：若 Claude Code 可通过 MCP 访问无头浏览器（Playwright / Puppeteer），信息采集是否会显著改善？

**WebFetch 的局限**：
- 纯文本抽取，丢失页面结构和视觉上下文
- 无法执行 JS，对 SPA 和动态加载页面无效
- 无法交互（翻页、搜索框输入、下拉加载）
- 部分网站反爬，WebFetch 直接返回空白或验证页面

**浏览器可补充的能力**：
- 动态页面渲染（SPA、JS 加载内容）
- 截图留存——某些信息的价值在视觉证据（如企业信息公示页面的截图比文字更权威）
- 交互操作——搜索框中输入、翻页、下拉加载
- PDF 在线预览时直接截图或提取文本

**但浏览器不是替代 WebFetch 的理由**：
- 慢——每个页面是完整渲染
- 重——token 消耗更大（Agent 需要解析页面结构/截图）
- 多数公开信息页面 WebFetch 就够用

**结论**：否决。当前数据源对 WebFetch 足够友好，三个实际瓶颈（搜索词质量、消歧、维度决策）均与浏览器无关。浏览器解决的不是真正卡住流水线的问题。

## 工作流范式：采集与报告的认知分离

**背景**：最初设计是每轮"搜索 → 入库 → 分析 → 决策下一步"的循环。实践后发现，调查中的信息处理实质上有两种不同的认知模式——采集和叙事——它们不应该被挤在同一轮里。

**采集阶段**（种子轮 + Round 1-N）：
- 每轮搜索由上一轮的分析结论导航，逐步逼近真相
- 分析 agent 的职责是为下一轮搜索提供方向（"接下来搜什么最可能改变认知"），不是为报告准备素材
- 采集轮的分析是 steering intelligence，不是 report material

**报告阶段**：
- 单独分派 report-agent，做跨轮合成
- report-agent 不重新分析原始数据，只基于分析 agent 已提炼的结论合成叙事
- 这是 narrative synthesis，不是 data reorganization

**种子轮的特殊性**：种子轮和其他轮唯一的区别是没有上一轮分析可参考，搜索方向完全由种子信息和调查目标决定。结构上和其他轮完全一致——搜索 → 入库 → 分析。

**待解决的分歧**：采集轮中分析 agent 和下一轮搜索应该串行还是并行？
- 并行（当前设计）：分析 agent 和下一轮搜索同时启动。更快，但下一轮搜索无法使用本轮分析结论作为导航——相当于隔一轮才用上分析结果。
- 串行：先分析再搜索。更准，但每轮多等一次分析的耗时。
- 中间路线：先把缺口驱动的机械任务搜起来，等分析回来后补搜分析引导的方向。

**原始信息归档**：分析 agent 在当前轮次产出后，追加写入 `证据日志.md`，记录"从哪知道了什么 + 关联节点/边 ID + 值得下载的文件"。主 Agent 根据日志中的建议下载文件到 `evidence/` 目录。不需要新增 MCP 工具或数据库表。

## Agent 分派时禁止 `isolation: "worktree"`

**问题**：子 Agent（web-search-agent、analysis-agent、report-agent）通过 `Agent` 工具分派时，若传入 `isolation: "worktree"` 参数，Claude Code 要求工作区为 git 仓库才能创建 worktree。调查工作区不一定是 git 仓库，导致报错。

**错误现象**：
- 子 Agent 无法启动
- 日志出现 git init 相关错误

**解决方案**：
- 所有子 Agent 分派时不得传入 `isolation` 参数。
- 子 Agent 在当前工作区进程内直接执行，不需要 worktree 隔离。
- 此约束需在 `SKILL.md` 核心约束和 `workflow.md` 步骤 3（分派子 Agent）中显式强调。

**当前状态**：`workflow.md` 步骤 3 已有记录（`不要传入 isolation: "worktree" 参数`），但 `SKILL.md` 的核心约束中尚未列出。应补充到核心约束中，因为这是一个高频操作错误。
